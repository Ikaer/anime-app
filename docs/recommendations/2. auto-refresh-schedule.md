# Scheduled auto-refresh

## What
Recompute the recommendations feed (the expensive fetch step, not the cheap
ranking) automatically on a schedule, instead of only on manual button press.

## Why
Today `performRecommendationsRefresh` in
[src/lib/recommendations.ts](../../src/lib/recommendations.ts) only runs from
`RecommendationsSection`'s "Rafraîchir" button via SSE
(`/api/anime/recommendations/refresh`). Crowd edges and MAL suggestions go
stale as soon as new completions/scores enter the list, and there's no
prompt to refresh — the feed silently drifts until someone remembers to
click. The app already has the exact infrastructure for this pattern:
`cron-sync.ts` fires `big-sync` + `performHistoricalCrawl` on a
`CRON_SECRET`-authenticated cron hit.

## Rough approach
- Extend `cron-sync.ts` (or add a sibling `cron-sync-recos.ts` for
  independent scheduling) to also call `performRecommendationsRefresh` after
  the existing big-sync/historical-crawl steps, using the last-known
  `nicheMode`/`threshold` from `getRecommendationsData()` rather than
  request params (there's no user present to supply them).
- Guard with the existing `isRecommendationsRefreshRunning()` lock so a cron
  tick never collides with a manual refresh.
- No SSE needed for the cron path — it's fire-and-forget like the rest of
  `cron-sync`; just log start/result.
- Optional follow-up: surface `lastRefresh` staleness in
  `RecommendationsSection` ("dernière actualisation il y a 6 jours") so it's
  visible whether the cron is actually running, without needing this as a
  hard dependency.

## Touch points
`src/pages/api/anime/cron-sync.ts`, `src/lib/recommendations.ts` (no new
logic — reuses `performRecommendationsRefresh` as-is), external: whatever
triggers the NAS cron job today for the existing cron-sync endpoint.
